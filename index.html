Technical Manual

1. Introduction and Software Architecture

TrackLift-Pro is a cross-platform desktop application built using the Electron framework by vibe coding with Antigravity IDE. The purpose of this software is to assist in railway maintenance by optimizing the vertical track profile for improving the running. By packaging Node.js backend capabilities alongside a Chromium frontend, the application provides a seamless desktop experience without the need for complex server deployments.

The user interface is constructed using standard web technologies: HTML5, CSS3, and JavaScript (ES6+). Styling is accomplished using vanilla CSS, ensuring a highly responsive, performant, and modern aesthetic without relying on heavy external framework dependencies. All data processing and intensive mathematical algorithms are executed locally on the client side via the V8 JavaScript engine, resulting in fast, native-like performance even when processing thousands of track stations.

2. Mathematical Foundations & Algorithms

The core engine of TrackLift-Pro relies on advanced heuristic optimization algorithms to compute the smoothest possible vertical track profile while respecting real-world lift and lower constraints. The software tries to bring the proposed levels within the permitted lift/lower and the unevenness/SD limits. After the optimization, the software mathematically drapes a smooth Cubic Spline over those points (Proposed rail levels) to export the continuous curved proposed rail profile.

2.1 Unevenness(V) Calculations

Unevenness or vertical versine is defined as the offset of the track from a theoretical chord connecting two points. In order to calculate the unevenness, short chord of 20m length and long chord of 80m length is used. The 20m/80m chords against 9m/18m chords in TRC will give much better analysis about the track condition. The default limits for unevenness for short and long chord are 5mm and 40mm respectively. These values are much strict than the New Track Tolerances as per IRPWM 2024 para 520.

2.2 Standard Deviation (SD) Calculations

Standard deviation represents the un-evenness of the track over a running window. In this software we adopted 200m rolling block for calculating the SD. i.e. for calculating the SD for a particular station, the unevenness values of 100m behind the station and 100m ahead the station is used. This continuously rolling block, unlike the fixed block for SD calculation in TRC, will give much realistic values. The default limits for SD values for short and long chord are 2mm and 20mm respectively.

In short, both the default Unevenness/SD value limits for short/long chords are much strict than the New Track Tolerances as per IRPWM 2024 para 520. Hence the proposed rail profile will offer a satisfactory riding experience. However, the user is having freedom to relax these limits as per site conditions.

2.3 Optimization Engine (Simulated Annealing)

The solver employs a heuristic optimization algorithm inspired by Simulated Annealing. This is a probabilistic optimization algorithm used to find the global maximum or minimum of a complex function over a large search space. It mimics the metallurgical process of heating and slowly cooling a material to minimize structural defects and achieve a stable state of minimum energy.

In the context of TrackLift-Pro, the 'cost function' is a mathematical penalty score representing how 'bad' the track geometry is (high versine, high standard deviation, or excessive lift/lowering). The goal of the engine is to find the track profile with the lowest possible cost. The detailed workflow is as follows:

  • Phase 1: Initialization - The system starts by copying the 'Existing Level' as the initial 'Proposed Level'. A virtual 'Temperature' is initialized at a high value. The initial Global Cost is calculated based on versine limit violations, SD violations, and lift/lower penalties.
  • Phase 2: Iterative Randomization - The engine selects a random, non-locked station and applies a tiny, random adjustment (up or down) to its Proposed Level, constrained by the Custom Max Lift/Lower limits. The software recalculates the entire track's Global Cost.
  • Phase 3: Acceptance Probability - If the new cost is LOWER (the track got smoother), the adjustment is immediately ACCEPTED. If the new cost is HIGHER (the track got worse), the engine uses the current 'Temperature' to calculate an acceptance probability. This allows the algorithm to escape 'local minima'.
  • Phase 4: Cooling Schedule - As iterations progress, the virtual 'Temperature' slowly cools down (decreases). The engine transitions from wild exploration into precise, localized fine-tuning.
  • Phase 5: Finalization - Once the temperature reaches zero or the max iterations are hit, the final Proposed Levels are locked in.

2.4 Lift vs. Lower Bias Weightage

In railway maintenance, lifting the track with ballast is structurally sound and standard practice. Lowering the track requires removing core ballast, which is a tedious job. The software is designed so that the solver explicitly hardcodes a 10x penalty multiplier to any 'lowering' adjustments. This mathematically biases the solver to overwhelmingly favour lifting the track at a 10:1 ratio.

2.5 The Hybrid Engine: Cubic Spline & Linear Interpolation

In order to make this software handy for field units during execution, an additional tamping PDF report can be generated, which gives lift/lower values at every 3.33m intervals. TrackLift-Pro utilizes a unique Hybrid Engine:
• Proposed Profile (Cubic Spline): A Natural Cubic Spline algorithm is applied to mathematically trace the physical elastic bending of the rail curve, outputting micro-precise 3.33m intermediate marks.
• Existing Profile (Linear Interpolation): To prevent artificial 'overshoots' and phantom potholes in the jagged existing profile, stable Linear Interpolation is actively used.

3. User Interface Guide

  • Project Details: Optional metadata fields (Zone/Division/Chainage) that automatically print on PDF and Excel exports.
  • Interval Rule: For the 20m and 80m chord checks to be valid, your station interval MUST be a clean divisor of 10m (e.g. 10m, 5m, 2m, 1m).
  • Smoothness Emphasis Factor:
    • Low (Weight=1): Minimizes track lifting. Allows small humps to remain.
    • Medium (Weight=10): Standard balanced smoothing.
    • High (Weight=50): Aggressively flattens humps by pushing track up to max lift.
    • Extreme (Weight=200): Overwhelming penalty to any defect. Relentlessly lifts track to achieve glass-like surface.
  • Max Lift/Lower: Global boundaries. Local overrides can be provided directly in the data table.
  • Lock Checkbox: Tick the 'Locked' box to force the optimizer to maintain the exact existing level at that point. The first and last stations are always auto-locked to avoid lifting at start/end stations.

4. Technical Constraints & Limitations

4.1 Virtual Boundary Extrapolation

To compute the unevenness with a 20m chord, 1 station behind and 1 station ahead is required. This is not met for the first/last stations. The software utilizes linear extrapolation to project gradients into 'virtual space' for computation purposes. These virtual calculations are hidden from the UI to prevent confusion.

4.2 Fixed SD Block Exception

Standard Deviation is calculated over a 200m "rolling window". For the first 100m, the software "locks" the window to evaluate the absolute first 200m chunk of the track (the SD for these stations will be identical). Similarly, the last 100m locks to the final 200m chunk. The middle section reverts to the dynamic, rolling window centered on each station.
